# 测试

在**仓库**内按项目现有约定执行；不要在工作区根目录臆造测试路径。

## 默认原则

- **业务逻辑变更**：应带**单元测试**（或项目规定的最小自动化验证），覆盖关键分支与边界。
- **纯重构、无行为变化**：以现有测试绿为主；必要时补回归用例。

## 与项目对齐

1. 读 `README`、`CONTRIBUTING`、CI 配置或 `pytest.ini` / `jest.config` 等，确认框架与命令。
2. 新测试放在与团队一致的目录（如 `tests/`、`__tests__/`、与源码同级的 `*.test.ts`）。

## 命名与组织

- 用例名说明**行为或预期**，而非仅方法名。
- 每个测试独立；共享用例使用 fixture / factory，避免隐式依赖顺序。

## 渐进增强

| 阶段 | 内容 |
|------|------|
| 起步 | 单元测试覆盖核心逻辑 |
| 中期 | API/模块级集成测试、关键路径 |
| 长期 | 冒烟/端到端（按团队投入与 CI 能力） |

## 提交前

在仓库根目录执行项目约定的测试命令（如 `pytest`、`npm test`、`pnpm test`），并报告失败时的关键错误信息。
