# 开发计划与架构决策

**大改动、多文件、跨模块或高风险变更**：先写 Plan，**用户确认后再写代码**。小范围 bugfix 可直接修。

## Plan 模板（可直接粘贴到对话或 `memory/`）

```markdown
## 背景
- 要解决什么问题 / 关联需求或 issue

## 目标与非目标
- 目标：
- 明确不做：

## 方案
- 首选方案（步骤、涉及文件/模块）
- 备选方案（简要）

## 影响范围
- 数据/接口/部署/兼容

## 风险与缓解
- 风险：
- 回滚方式：

## 验证
- 测试命令 / 手工检查清单

## 待确认问题
- （列出需要用户拍板的事项）
```

## ADR（架构决策记录）精简模板

```markdown
# ADR-NNN: 标题

## 状态
提议 / 已采纳 / 已废弃

## 背景
## 决策
## 后果（正面与负面）
```

重要决策可落盘到仓库的 `docs/adr/` 或团队约定路径，并在 `MEMORY.md` 里记一条索引。

## 确认后再执行

- 用户确认 Plan 或逐条回答「待确认问题」后，再开始改代码与提交。
