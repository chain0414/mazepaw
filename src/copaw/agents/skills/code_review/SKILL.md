# 代码评审

在提交或请求他人评审前，先做**结构化自检**；输出时保持一致粒度，避免「一轮只提一半」。

## 评审维度

1. **正确性** — 行为是否符合需求与边界条件
2. **安全** — 注入、鉴权、敏感数据、依赖漏洞面
3. **可维护性** — 命名、模块边界、重复与抽象是否合适
4. **性能** — 明显热点、N+1、无谓分配
5. **测试** — 关键路径是否有覆盖；回归场景是否想到

## 严重级别（建议在评论中标注）

| 标记 | 含义 |
|------|------|
| 阻塞 | 必须修复才能合并（安全、数据损坏、破坏 API 契约等） |
| 建议 | 应修复或强烈建议（缺失校验、重要路径无测试等） |
| 备忘 | 可选优化、风格、后续重构 |

## 单条评论格式示例

```
**阻塞 · 安全**：`query` 直接拼接用户输入，存在 SQL 注入风险。
建议：`db.query(..., params=[user_id])` 或使用 ORM 参数绑定。
```

## 何时自评即可

- 极小改动且已本地跑通测试与静态检查
- 用户明确只要快速草稿

## 何时应请人审

- 安全相关、权限模型、支付/数据迁移
- 大范围重构或公共 API 变更
