## 仓库优先

- **代码与 Git 的默认范围**在 `REPO.md` 中列出的**本地仓库路径**（`repo_assets`），不是本 agent 工作区根目录。
- 用户说「仓库」「代码」「项目」「改这个库」时，先在绑定仓库路径下读代码、跑命令；工作区目录只放记忆、配置与技能。
- 多仓库时：先确认操作的是哪一个（名称 / ID），再切换工作目录。

## 代码变更协议

- **大改动**：先产出可执行的 Plan（背景、方案、影响面、风险、回滚），**用户确认后再改代码**。详见 `active_skills/dev_planning/`。
- **每次修改**：尽量在回复中说明改了哪些文件；能展示 diff 或要点摘要时优先展示。
- **业务逻辑**：新增或变更行为时，**配套单元测试**（或项目约定的最小测试）；见 `active_skills/testing/`。
- **破坏性操作**（删文件、强推、改历史）：先征得同意。

## Git 与分支

- 动手改代码前，在目标仓库内 **fetch/pull**，了解当前分支与远端状态。
- 小改动可直接在约定分支上工作；**跨模块、高风险或长时间**的改动，单独开 feature 分支，验证后再 **MR/PR** 合入主干。流程细节见 `active_skills/git_workflow/`。

## MR 与待审核队列

若团队使用 CoPaw **待审核队列**（控制台「待审核队列」或 `POST /insights/review-queue`）作为合并前门禁：

- **先**在队列里提交一条 **merge_request** 类型说明（标题、摘要、可选 MR 链接），**用户在队列中批准**后，再指导用户在 GitHub/GitLab **Approve / Merge** MR。
- **不要**在队列未批准前，默认可以合并或替用户完成合并决策。
- 队列条目写入方式见 `active_skills/git_workflow/` 中「MR 与待审核队列」。

## 代码审查（含自检）

- 合并或交付前，按维度自检：**正确性、安全、可维护性、性能、测试覆盖**。
- 输出可遵循：阻塞项 / 建议 / 备忘；见 `active_skills/code_review/`。

## 记忆（开发向）

每次会话独立；持久上下文在工作区文件中：

- **每日笔记：** `memory/YYYY-MM-DD.md`
- **长期记忆：** `MEMORY.md` — 架构决策摘要、技术选型、反复出现的坑、与仓库相关的约定
- 更新前先 `read_file` 再写入，避免覆盖。

值得主动写入的内容示例：ADR 结论、模块边界、性能基线、与业务强相关的 invariant、Code review 中达成的规范。

## 安全

- 不泄露密钥、令牌、内网地址等敏感信息到无关输出。
- 不确定是否安全时，先问用户。

## 工具

需要专项流程时，查看对应 `active_skills/<name>/SKILL.md`。

## Heartbeat

若配置了 heartbeat：有 `HEARTBEAT.md` 则严格按其清单执行；勿臆测历史会话中的未完成任务。
